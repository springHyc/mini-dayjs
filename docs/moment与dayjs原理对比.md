# moment 与 dayjs：原理与架构对比

> **配套阅读**：dayjs 通用原理见 [`dayjs原理概述.md`](./dayjs原理概述.md)；与 mini-dayjs 代码对照见 [`dayjs原理与mini-dayjs对照.md`](./dayjs原理与mini-dayjs对照.md)。

---

> 说明：二者都是 **「日历/时间」工具库**，底层都 **依赖宿主环境的时间能力**（主要是 **`Date`**，moment 内部还有自己的解析与字段组织），**都不是**从零实现格里高利历。下面从 **原理与产品形态** 对比，方便分享时一张表讲清「为什么 antd 4 用 moment、5 换 dayjs」背后的工程取向。

---

## 1. 共同点（讲稿可一句带过）

- **目标类似**：解析输入、加减、对齐（`startOf`）、格式化、locale、时区相关能力（各自实现路径不同）。
- **都不替代 `Date` 的全部语义**：最终仍要与 **系统时区、夏令时规则、引擎实现** 打交道；复杂 IANA 时区场景两边都要 **额外能力**（moment 有 `moment-timezone`；dayjs 有 `utc` + `timezone` 插件）。

---

## 2. 架构：「全家桶」vs「核心 + 插件」

| 维度             | moment                                                                                                      | dayjs                                                                                         |
| ---------------- | ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **默认安装形态** | **单一大模块**：常用 API、locale、多类能力 **默认随包体思路偏「一体化」**（历史上打包体积大是主要槽点之一） | **仅核心**：文档写明 **默认不包含任何插件**；能力拆到 **`dayjs/plugin/*`**，**按需 `extend`** |
| **扩展方式**     | 以 **同一命名空间** 挂载为主（插件也存在，但整体仍偏「一大块」）                                            | **物理分包**：插件与核心 **文件级分离**，利于 **tree-shaking** 与 **按需引入**                |
| **原理含义**     | 更像 **「一站式库」**：功能集中，心智简单，**代价是默认体积**                                               | 更像 **「内核协议 + 可选扩展」**：默认薄，**代价是** 要记得 **引插件、引 locale**             |

---

## 3. 可变性与实例语义

| 维度           | moment                                                                                                   | dayjs                                                                                     |
| -------------- | -------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **经典用法**   | **`add` / `subtract` 等常表现为「可变」**：同一实例在链式调用中被 **原地更新**（除非显式 **`clone()`**） | **偏不可变**：`add` / `startOf` 等通常 **返回新实例**，减少 **共享引用被悄悄改掉** 的问题 |
| **原理层理解** | **可变模型** 利于链式写法的性能与写法惯性，但 **隐式副作用** 在复杂组件里更难排查                        | **新实例模型** 与 React / 函数式 **数据流** 更合拍；实现上多是 **拷贝再改**               |

（分享时可各举一例：`moment` 链式后同一变量是否被改；`dayjs` 两次 `add` 是否互不影响。）

---

## 4. 解析与格式化在「内核」中的比重

| 维度       | moment                                                                             | dayjs                                                                       |
| ---------- | ---------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **解析**   | **自定义格式解析**、严格模式等 **深度集成** 在整体能力里，**体感是「默认就很全」** | **基础解析** 在核心；**严格按格式解析** 多依赖 **`customParseFormat` 插件** |
| **format** | **大量 token / locale 行为** 随库一体出现（体积来源之一）                          | **核心 `format` 较薄**；更多占位符常靠 **`advancedFormat`** 等插件          |

**原理结论**：不是「谁能解析更多字符串」，而是 **解析与格式化的代码放在「默认 bundle」里还是「插件文件」里** 的产品取舍。

---

## 5. 体积与打包（前端最关心的「原理后果」）

| 维度            | moment                                                                          | dayjs                                                               |
| --------------- | ------------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| **默认 bundle** | 历史上 **显著更大**；社区常见做法是 **不打包进业务** 或 **换 dayjs / date-fns** | **默认只含核心**，再按页/按功能加插件，**gzip 后通常远小于 moment** |
| **工程原理**    | **一体化** → 用一点也易拖进一大片                                               | **分层** → **未 import 的插件不进包**（在构建配置正确的前提下）     |

---

## 6. 生态与维护状态（分享可提一句）

| 维度         | moment                                                | dayjs                                                    |
| ------------ | ----------------------------------------------------- | -------------------------------------------------------- |
| **维护策略** | 官方进入 **维护模式**：**不再加新能力**，以修问题为主 | **仍在积极迭代** 功能与插件（具体以仓库为准）            |
| **与 UI 库** | **antd ≤4** 等历史选型常绑定 **moment**               | **antd ≥5** 默认 **dayjs**，减轻体积与 tree-shaking 压力 |

---

## 7. 一句话对照（适合 PPT 收尾）

- **moment**：**一体化、功能默认很全、可变链式为主**，适合当年「一把梭」；**代价是默认体积与隐式可变**。
- **dayjs**：**核心薄、能力插件化、实例语义偏不可变**，适合现代打包与组件数据流；**代价是** 要养成 **按需 extend + locale** 的习惯。

---

## 8. 总表（速查）

| 对比项          | moment                        | dayjs                      |
| --------------- | ----------------------------- | -------------------------- |
| 默认包体取向    | 大、一体化                    | 小、仅核心                 |
| 能力扩展        | 库内一体为主                  | **插件文件 + `extend`**    |
| 实例可变/不可变 | **可变** 为主（`clone` 规避） | **新实例** 为主            |
| 与 `Date` 关系  | 封装 + 自有解析管线           | **薄封装 `$d`** + 插件补全 |
| 维护            | 维护模式                      | 活跃迭代（以仓库为准）     |
| 典型选型动因    | 历史项目、API 习惯            | 体积、antd5、按需加载      |
